# RAG 知识库问答系统 — 功能与技术介绍

## 项目整体定位

这是一个基于 **RAG（检索增强生成）** 架构的企业知识库问答系统。核心思路是：把文档"喂"给系统，然后用自然语言提问，系统从文档里找答案，而不是靠 AI 自己"编"，从根本上解决大模型"幻觉"问题。

---

## 功能模块详解

### 1. 文档上传与处理

**功能**：用户上传 PDF / Word / TXT，系统自动解析文本、切块、向量化存储。

**技术实现：**

| 步骤 | 技术 | 说明 |
|------|------|------|
| 文件解析 | PyPDF2 / python-docx | 分别处理 PDF 和 Word 文档 |
| 文本分块 | 滑动窗口切块 | Chunk Size=800、Overlap=200，保留块间上下文 |
| 向量化 | 智谱 Embedding-3 | 每个文本块转为 1536 维向量 |
| 存储 | ChromaDB | 本地向量数据库，附带文件名、页码等元数据 |

---

### 2. 智能问答（RAG 核心）

**功能**：用户输入问题，系统从知识库检索最相关文档片段，交给大模型生成答案，并返回引用来源和相似度评分。

**完整处理链路：**

```
用户提问
  → 问题向量化（Embedding-3）
  → 向量相似度检索（ChromaDB，Top-K=3）
  → 构建 Prompt（问题 + 检索到的文档片段）
  → 调用智谱 GLM-4-air 生成答案
  → 返回答案 + 来源文档（文件名 + 相似度分数）
```

**关键设计：**
- API 端点：`POST /api/query`，由 FastAPI 处理
- 来源追溯：每条 AI 回复都附带引用了哪些文档片段及相似度分数，答案可验证
- Prompt 工程：严格约束 LLM 只基于检索到的文档回答，不知道则明确告知

---

### 3. 多轮对话与会话管理

**功能**：用户可以创建多个会话、切换历史对话，对话记录持久化不丢失；AI 能理解上下文，支持追问（如"它是什么意思"）。

**后端实现（数据持久化）：**

- 引入 **SQLite + SQLAlchemy ORM**，设计两张表：

```sql
-- 会话表
conversations (id, title, created_at, updated_at, is_deleted)

-- 消息表
messages (id, conversation_id, role, content, sources JSON, created_at)
```

- 新增 7 个 RESTful API 端点：

| 方法 | 路径 | 功能 |
|------|------|------|
| POST | `/api/conversations` | 创建会话 |
| GET | `/api/conversations` | 获取会话列表（分页） |
| GET | `/api/conversations/:id` | 获取会话详情 |
| PUT | `/api/conversations/:id` | 更新标题 |
| DELETE | `/api/conversations/:id` | 软删除 |
| GET | `/api/conversations/:id/messages` | 获取消息列表 |
| POST | `/api/conversations/:id/query` | 带上下文的 RAG 问答 |

- **多轮上下文**：每次问答取最近 10 条历史消息拼入 Prompt，LLM 能理解代词指代和追问逻辑
- **自动标题生成**：首条消息自动截取前 30 字作为会话标题

**前端实现（状态管理）：**

- 使用 **Zustand** 做全局状态管理，集中维护会话列表、当前会话 ID、消息列表
- 新增 `ConversationSidebar` 组件：会话列表展示、新建、内联重命名、删除二次确认
- **乐观更新**：发送消息时先在本地立即展示用户消息，再等待 AI 响应插入，减少等待感
- **自动容错**：发送失败时回滚消息，若会话为空则自动清理，避免产生垃圾数据

---

### 4. 知识库管理

**功能**：查看已上传的所有文档，包括文件名、大小、上传时间、分块数量，支持单个删除。

**技术实现：**
- 文档元数据存储在 ChromaDB 的 metadata 字段中
- `GET /api/documents` 返回全部文档列表
- `DELETE /api/documents/:id` 从向量库中删除对应文档的所有向量块
- 前端使用 Ant Design Table 组件展示，支持文件大小格式化显示

---

### 5. 前端整体界面

**功能**：三页面 SPA，分别为智能问答、知识库管理、系统概览，聊天区支持 Markdown 渲染。

**技术实现：**

| 技术 | 用途 |
|------|------|
| React 18 + TypeScript | 前端框架，类型安全 |
| Vite | 构建工具，开发热更新 |
| Ant Design | UI 组件库（表格、弹窗、表单等） |
| React Router | 客户端路由，三个页面 |
| react-markdown | 渲染 AI 回复中的代码块、列表、标题 |
| Axios | HTTP 客户端，统一封装所有 API 调用，60s 超时 |
| Zustand | 轻量级全局状态管理 |
| dayjs | 日期格式化（相对时间显示） |

- 类型安全：`types/api.ts` 中定义的 TypeScript 接口与后端响应结构完全对应
- 渐变主题：统一使用 `#667eea → #764ba2` 紫色渐变配色

---

### 6. 后端服务架构

**功能**：提供统一的 REST API，连接大模型、向量库、关系数据库三个数据源。

**技术实现：**

- **FastAPI**：高性能异步框架，自动生成 OpenAPI 文档（访问 `/docs` 即可调试）
- **分层架构**：

```
api 路由层（conversation_routes.py / routes.py）
    ↓
service 业务层（conversation.py / vectordb.py / document.py）
    ↓
core 核心算法层（rag.py / embeddings.py / llm.py）
```

- **依赖注入**：FastAPI `Depends(get_db)` 自动管理数据库 Session 生命周期
- **配置管理**：`pydantic-settings` 统一读取 `.env` 环境变量（API Key、模型名、分块参数）
- **CORS**：中间件允许前端跨域访问
- **日志**：`loguru` 记录每次请求的关键步骤，格式化输出

## 一句话总结

| 功能模块 | 核心技术链路 |
|----------|-------------|
| 文档上传 | PyPDF2/python-docx 解析 → 切块 → Embedding-3 向量化 → 存 ChromaDB |

| 前端界面 | React + TypeScript + Ant Design，Markdown 渲染，Axios 通信 |
